TroubleShooting a Users HomeDrive Auto-Remapping

Issue: 

  • Four users report that their ‘HomeDrive’ mapping is defaulting to Z:\ during login.
  • Desktop Team reports that even after mapping and remapping the drive to N:\ within windows the drive reverts back to Z:\
  • Active Directory Users and Computers shows that the ‘HomeDrive’ set to N:\ on the profile tab of the AD User Object
  • Desktop Team suspects that a GPO is making changes to the drive mappings for these users.

Initial Troubleshooting Efforts: 

Scoured the (local and global) GPO’s to make absolutely sure that nothing was remapping the HomeDrive.

With PowerShell perform a check of the users: * Group Memberships * Organizational Units * Time frame that the effected users were created.

Compare these results with that of other users created in this time frame, and all other users in the domain.

1

Get-ADUser $UserWithIssue -Properties *

Initial Findings: 

These four users were in the same Organizational Unit as every other user in the domain, and had exactly the same Group Memberships as six others that were created within the same time period.

Initial Conclusion: 

This is not an issue with a GPO. GPO would have produced a consistent and widespread experience across the user base.

Next Steps: 

I realize that it may be time to ask for some help.

I explain the situation to a co-worker and they remembered working on an issue a few months earlier where the user had an invalid “HomeDrive” setting on their AD Object. The ADU&C GUI showed it properly mapped to “N:” but PowerShell exposed the HomeDrive property was actually “N” - with no colon at the end.

A check of the effected users AD Object in PowerShell revealed that it indeed was only “N”

Root Cause Analysis 

A change to the ‘New User Creation’ script inadvertently removed the ( : ) colon where the drive letter was assigned. Examination of the code uncovered that if the Location drop down was set to Cincinnati the -HomeDrive property was set to be ‘N’ …without the colon, where as the code to set other locations were properly being set as ‘N:’

Final Conclusion: 

The Active Directory property ‘HomeDrive’ requires a ( : ) colon after the drive letter to be valid. When encountering an invalid entry, the HomeDrive will default to the last available drive letter.

Running the command:

1

Get-ADUser -Filter {Homedrive -eq 'N'} -Properties Homedrive

This output revealed ~160 other users that also had this issue associated with their account. No idea why none of these others had not reported problems.

Since ‘N’ was an invalid drive setting, windows defaulted to using the last available drive letter and assigned that as the HomeDrive.

Example of the error in the code:  

1
2
3
4

if ($LocationCombo.SelectedItem -like " Cincinnati*") 
{ 
    $CommandOutput += " -HomeDrive ""N"" -HomeDirectory ""\\DFSRoot\BaseBall\FansInOH\$($Usernamebox.Text)"""
}

Correction of the error in the code:  

1
2
3
4

if ($LocationCombo.SelectedItem -like " Cincinnati*") 
{ 
    $CommandOutput += " -HomeDrive ""N:"" -HomeDirectory ""\\DFSRoot\BaseBall\FansInOH\$($Usernamebox.Text)"""
}

Remediation 

First we needed to find all the users who also had this issue.

How to find the users effected by this issue:  

1
2

# This will find users regardless of the drive letter used:
Get-ADUser -Filter {Homedrive -notlike '*:'} -Properties Homedrive | Where-Object {$_.Enabled -eq $true} | Select-Object Name, HomeDrive

- or -

1
2
3

# Capture all users with N as the drive letter with no colon present
# into the variable $NDriveUsers for use in the remediation
$NDriveUsers = Get-ADUser -Filter {Homedrive -eq 'N'} -Properties Homedrive

Correct users effected by this issue  

1

$NDriveUsers | Set-ADUser –HomeDrive “N:”

Check an individual user  

1
2
3
4

foreach ($NDriveUser in $NDriveUsers.SamAccountName)
{
    Get-ADUser $NDriveUser -Properties Homedrive  | Select-Object Name, HomeDrive
}



TL;DR: 

If you have a user whose HomeDrive mapping is changing to something other that what it should be set to within Active Directory, check that users AD Object, as it may have an invalid property setting.

Correct your new user creation scripts, users whose HomeDrive is set through the ADU&C GUI users do not have this issue.

Finally, share your experiences and don’t be afraid to ask for help… once I evoked the assistance of my teammate the issue was isolated and resolved within 30 minutes… including root cause analysis, remediation of user accounts, and correction of the root cause.
